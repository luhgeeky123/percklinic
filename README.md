# percklinic
Мой проект «База данных поликлиники» представляет собой учебную реляционную базу данных, спроектированную для моделирования работы медицинской клиники. Проект включает в себя:
Нормализованную схему данных в третьей нормальной форме (3НФ), охватывающую сущности: пациенты, врачи, специализации, приёмы, диагнозы, лекарства, производители лекарств и назначения.
Полный SQL-скрипт, содержащий:
создание всех таблиц с внешними ключами,
наполнение данных (реальные примеры пациентов, врачей, лекарств и визитов),
Таблицы:
1. Specialization — Специализации врачей
Хранит справочник медицинских специальностей.
Поля:
specialization_id — уникальный идентификатор специализации (первичный ключ),
name — название специализации (например, «Терапевт», «Кардиолог»).
Обеспечивает нормализацию: избегаем дублирования названий специальностей у врачей.

2. Proizvoditel — Производители лекарств
Справочник фармацевтических компаний.
Поля:
proizvoditel_id — уникальный ID производителя,
name — название компании (например, «Pfizer», «Фармстандарт»),
country — страна происхождения.
Вынесено в отдельную таблицу для соблюдения 3НФ: лекарства зависят от производителя, а не хранят его данные напрямую.

3. Patient — Пациенты
Информация о пациентах клиники.
Поля:
patient_id — уникальный ID пациента,
full_name — ФИО,
date_of_birth — дата рождения,
gender — пол («M»/«F»),
phone, email, address — контактные данные.

4. Doctor — Врачи
Данные о медицинском персонале.
Поля:
doctor_id — уникальный ID врача,
full_name — ФИО,
specialization_id — ссылка на таблицу Specialization,
phone, email, cabinet_number — контакт и кабинет.
Связь «многие к одному»: несколько врачей могут иметь одну специализацию.

5. Visit — Приёмы (визиты)
Фиксирует факт визита пациента к врачу.
Поля:
visit_id — уникальный ID приёма,
patient_id — ссылка на пациента,
doctor_id — ссылка на врача,
visit_datetime — дата и время приёма,
status — статус («запланирован», «завершён», «отменён»).
Центральная связующая таблица между пациентами и врачами.

6. Diagnosis — Диагнозы
Хранит медицинские заключения по каждому завершённому приёму.
Поля:
diagnosis_id — уникальный ID диагноза,
visit_id — ссылка на конкретный приём (уникальная, т.к. один визит — один диагноз),
description — текст диагноза,
diagnosis_date — дата постановки.
Связь «один к одному» с Visit.

7. Medication — Лекарства
Справочник медикаментов.
Поля:
medication_id — уникальный ID лекарства,
name — название (например, «Амоксициллин»),
proizvoditel_id — ссылка на производителя,
price — стоимость,
instructions — инструкция по применению.
 Все данные о лекарстве хранятся централизованно — соблюдена 3НФ.

8. Prescription — Назначения (рецепты)
Фиксирует, какие лекарства были назначены на конкретном приёме.
Поля:
prescription_id — уникальный ID назначения,
visit_id — ссылка на визит,
medication_id — ссылка на лекарство,
dosage — дозировка и способ применения.
Связь «многие ко многим» между визитами и лекарствами (через промежуточную таблицу).

Все таблицы связаны внешними ключами, не содержат избыточных данных и соответствуют третьей нормальной форме (3НФ).
Проект позволяет не только хранить данные, но и выполнять аналитические запросы для управления клиникой.
три аналитических запроса:
Доктор с наибольшим числом визитов за последний месяц,
ТОП-5 наиболее часто назначаемых препаратов,
Список пациентов с их диагнозами и именами врачей в формате JSON.
